Document de **cadrage et ordre de livraison** pour le socle organisationnel client et la chaîne **OWN / SCOPE / ALL** (permissions, résolution de scope, moteur d’accès, diagnostic, migration). Les états **État** reflètent le dépôt au moment de la dernière mise à jour documentaire (**2026-05**) ; source de vérité détaillée : chaque RFC et le code sous `apps/api/` / `apps/web/`.

## Ordre prioritaire recommandé

**État livré (données + API + socle RBAC + résolution scope + politique 017 + moteur 018 — pilote Projets lecture)** : **RFC-ORG-001**, **RFC-ORG-002 (MVP)** et **RFC-ORG-003 (V1)** sont en place pour rattacher les personnes HUMAN aux comptes client, exposer `ownerOrgUnitId` / résumés métier et auditer les changements d’ownership. **RFC-ACL-015** livre le **socle vocabulaire** : codes `*_own` / `*_scope` / `*_all` (et associés) seedés, contrôle unifié `satisfiesPermission` sur les guards. **RFC-ACL-016** livre le **service de résolution** : `OrganizationScopeService` calcule `ALL`/`OWN`/`SCOPE`/`NONE` + `reasonCodes` typés (BFS in-memory sur arbre actif, cache requête), **consommé** par **RFC-ACL-018** sur le périmètre **Projets** (lecture détail + liste). **RFC-ACL-017 (V1)** livre **`ResourceAccessPolicy`**, la matrice `DEFAULT` / `RESTRICTIVE` / `SHARING`, l’API liste ACL enrichie, `PATCH …/access-policy` et le **batch** `evaluateResourceAccessBatch` utilisé par le moteur. **RFC-ACL-018 (V1 backend)** : module `access-decision`, verdict `AccessDecisionResult`, **`sharingFloorAllows = floorAllowed`**, pas de pré-filtre org avant policy/ACL ; **Projets** : `assertAllowed` (read) + `filterResourceIdsByAccess` (list) ; intents **`write`/`admin`** sur le moteur et **autres modules** → **RFC-ACL-020**. **RFC-ACL-019 (V1)** : diagnostic enrichi **derrière le flag** `ACCESS_DIAGNOSTICS_ENRICHED` (`true` ou `1` au sens strict) — alignement **read** sur `decide`, blocs organisation / propriété / politique, `evaluationMode` sur les six couches ; sans flag, le contrat JSON reste celui d’origine (**011**/**014**).

La chaîne **RFC-ACL-017 → 018 → 019** suit une mise en prod progressive : politique d’accès d’abord, moteur unifié ensuite, diagnostic aligné sur ce moteur. **RFC-ACL-022** est avant **RFC-ACL-021** pour que backfill, rapports d’écarts et feature flags nourrissent un cockpit utile plutôt qu’une vue vide ou bruitée.

**RFC-ACL-020** et **RFC-ACL-022** : en delivery, les traiter comme **une même tranche par module métier** (ex. Projets, puis Budgets…) — pour chaque périmètre, livrer ensemble **flags + migrations/backfill (022)** et **branchement ownership, guards et filtrage liste (020)** ; éviter d’activer en production le nouveau comportement **020** sur un module sans le socle **022** correspondant sur ce même module.

| Ordre | RFC             | Pourquoi                                                                                    |
| ----: | --------------- | ------------------------------------------------------------------------------------------- |
|     1 | **RFC-ORG-002** | Lien `ClientUser` ↔ `Resource` HUMAN (données + API) — **MVP livré** ; sans ce socle, pas de résolution fiable `User → HUMAN` pour `OWN` / `SCOPE` |
|     2 | **RFC-ORG-003** | **V1 livrée** — `ownerOrgUnitId` (+ résumés, audits, UI minimale) : données nécessaires pour rattacher les ressources métier à une `OrgUnit` avant calcul `SCOPE` (**RFC-ACL-016**) |
|     3 | **RFC-ACL-015** | Les permissions `OWN / SCOPE / ALL` doivent exister avant le moteur                         |
|     4 | **RFC-ACL-016** | Le service de résolution du scope est le cœur métier du modèle cible                        |
|     5 | **RFC-ACL-017** | **V1 livrée** — politique par ressource + API + UI + batch ; matrice consommée par **018**            |
|     6 | **RFC-ACL-018** | **V1 (backend) livrée** — pilote **Projets** (lecture détail + liste filtrée) ; guard HTTP + autres entités → **020** ; diagnostic → **019**                                       |
|     7 | **RFC-ACL-019** | **V1 livrée** (flag `ACCESS_DIAGNOSTICS_ENRICHED`) — diagnostic **read** aligné sur le moteur **018**, blocs org / ownership / policy, UI matrice + popover |
|     8 | **RFC-ACL-020** | Intégration progressive **par module**, **couplée** à la même tranche **RFC-ACL-022** (flags + données) sur ce module |
|     9 | **RFC-ACL-022** | Migration, backfill et feature flags **par module**, **couplés** au branchement **RFC-ACL-020** sur ce module — pas d’activation 020 seule sans 022 |
|    10 | **RFC-ACL-021** | Cockpit de pilotage une fois les données et rollouts (020 + 022) disponibles                |
